\begin{theory}{Why prove algorithms?} \\
    Sometimes, its not obvious if an algorithm does the things its advertised through common sense:
    \begin{itemize}
        \item Will your algorithm terminate in a timely manner?
        \item Can you prove that the algorithm won't infinitely loop?
        \item Is the time complexity as advertised?
    \end{itemize}
    Proofs are needed for these cases, most of the time they're the \textbf{only} way to know if the algorithm does its job.
\end{theory}

\begin{example}{Stable matching example} \\
    There are \textit{n} hospitals and \textit{n} doctors. Each hospital wants to hire exactly one doctor. Each doctor and hospital submit a list of preferences ranking which hospital they want to go to and which doctor they want.

    How do we design an algorithm where there is no hospital doctor pair (\textit{h, d}) that are dissatisfied with their result (stable):

    \begin{center}
        for pair $p = (h, d')$ \& $p' = (h', d)$\\
        \begin{itemize}
            \item $h$ prefers $d$ over $d'$
            \item $h'$ prefers $d'$ ovr $d$
        \end{itemize}
    \end{center}
    This implies that both hospitals and doctors have a preferred swap to make (unstable
    match) \par
    \bigskip
    \noindent\textbf{Stable matching algorithm} \par
    \noindent While there is a free hospital, pick a free hospital $h$ and offer a job to the highest
    ranking doctor $d$ on its preference list. \par
    \bigskip
    \noindent If $d$ doesn't have a job yet, $d$ always accepts the job.\par
    \par
    else if $d$ is already in a pair $(h' d)$:
    \begin{itemize}
        \item{if $h$ is higher ranked on the preference list for $d$, a new pair $(h, d)$ is
            created and the previous pair $(h', d)$ is deleted.}
        \item{if $h$ is ranked lower, then the offer is ignored.}\par
    \end{itemize}

    \noindent\textbf{How many ways to match hospital-doctors?} \par
    \noindent $n!$, first doctor has $n$ choices, next doctor has $n-1$ choices, so on and so forth.
    \par

    \noindent\textbf{Will a stable match always exist?} \par
    \noindent Yes, but it is not obvious - we need to use a proof:\par
    \par
    note that during the while loop:
    \begin{itemize}
        \item{for a doctor, the hospital they go to will always be higher on their ranks.}
        \item{a hospital will be paired with doctors of decreasing ranks.}
    \end{itemize}
    If we assume that the ranking is not stable: $p = (h, d)$ and $p' = (h', d')$ such that
    $h$ prefers $d'$ and $h'$ prefers $d$.\par
    \par
    Since $h$ prefers $d'$ to $d$, then it must have made an offer to $d'$ before offering to
    $d$, but since $h$ is paired with $d$, then $d'$ would need to have rejected $h$ because
    tehy were already at a hospital they prefer to $h$. This proves that its impossible to
    have $d'$ prefer $h'$ if $h$ is ranked higher.
\end{example}

\begin{theory}{Big O Notation} \\
    We say that $f(n) = O(g(n))$ if: \\
        \indent There are positive constants $C$ and $N$ such that \\
        \indent $0 \leq{f(n)} \leq{C\times{g(n)}}$ for all $n \geq{N}$.\\
    \\
    What does that acutally mean? lets take a look at some examples.\\
    \\
    Is $f(n) = O(g(n))$ for: \\
    \indent$f(n) = n + 5$, $g(n) = n^2-1$\\
    \\
    For this to hold true, there needs to be a value $n$ and $C$ such that
    every single value of $f(n) \leq{g(n)}$. \textbf{Is this true?}\\
    \\
    Well, given a sufficiently large $n$, yes! Lets prove it:\\
    Lets assign $C = 1$\\
    \begin{gather}
    f(n) = n + 5 < g(n) = n^2 - 1\\
    0 < n^2 - n - 6\\
    0 < (n - 3)(n + 2)\\
    \therefore n \text{ is true for } n > 3 \text{ or } n < -2
    \end{gather}
    \\this implies that for values $C = 1$ and every value $n > 3$, this equation holds true!\\
    In more human terms, we \textbf{need} to have a value $C$ that is multiplied with $g(n)$, 
    and there \textbf{needs} to exist a value $n$ such that every value greater than $n$ will satisfy
    \[
        0 \leq f(n) \leq C \times g(n).
    \]
    In doing so, we can say that the for $f(n)$ (which is the function of input size vs time) has
    a time complexity of $O(g(n))$ (keep in mind $g(n)$ will just be an equation that $f(n)$ will never
    exceed.)\\

    Keep in mind that big O tends to overestimate the complexity (by virute of upper
    bounding at an asymtopic point).
\end{theory}

\begin{aside}{Is big-oh the be all-end all?}\\
    We know that big-oh demonstrates the worst case time complexities. But theres alot of
    data we're missing out on! What about best case? There are algorithms out there that
    can run faster than other algos even if they are bounded by the same big-oh time
    complexity! Think about running bubble sort on a sorted array with the first and last
    items swapped vs selection sort. This is where big theta and omega comes in.
\end{aside}

\begin{theory}{Big Omega}\\
    Big omega is the inverse of big-oh, it yields the asymtopic lower bound of a function.
    In plain words, $\Omega(f(n))$ is a metric that given a long enough running time,
    $f(n)$ will always be at least $\Omega(f(n))$. Another way to think about it is that
    given a time complexity $f(n)$, it will never be less than $\Omega(f(n))$ so plotting
    it on a graph will show that $\Omega(f(n))$ is bounded by $f(n)$.
\end{theory}

\begin{theory}{Big theta}\\
    given that we know $O(f(n))$ and $\Omega(f(n))$, we can combine those two graphs
    (where the space between big-oh and big omega represents big theta!). This means that
    the growth rate is the same in both best and worst case senarios.
\end{theory}

\begin{theory}{sum and product property}\\
    for all time complexities, we want to know how summing differnet time complexities and the product of them
    chagnes the total time complexity!

    for the sum of time complexities, we take the larger value: $$O(a) > O(b), O(a) + O(b)
    = O(a)$$. As an example, this makes sense! $$O(n) + O(n^2) = O(n^2)$$

    for the product of time complexities, the product of two time complexities will result
    in the product of the growth rates: $$O(a) \times O(b) = O(ab)$$.
\end{theory}

\begin{theory}{Proof by induction}\\
    Proof by induction is pretty fun. Generally, induction is used to prove a sequence of
    things (that are ordered). If we prove that given any position, the next one holds a
    certain property and the first item also holds that property, then we can extrapolate
    that the entire set holds the property. For instance, for an infinitely long set of
    stairs that are green, if we are given the property that at any green step, the next
    step will be green. \\

    There exists another kind of induction, strong induction. Normal indcution assumes
    something at a point: at $k$, the $k+1$th step is green. However for strong induction,
    we assume it is true from the base case to $k$, i.e. $1, 2, ..., k$th steps are
    all green. \\

    To prove for induction, we have to prove three things:
    \begin{enumerate}
        \item the base case is correct
        \item assume it is true for a $k$ (or up to $k$ if we are doing strong induction)
        \item prove it is true for the $k+1$th item
    \end{enumerate}
    An induction proof is not complete without these three properties.
\end{theory}

\begin{theory}{Proof by invarients}\\
    Proof by invarients is pretty similar to induciton: We wnat to prove that a property P
    holds true over an algorithm, and that if P is true at the start, and if we can prove
    that at any step we can preserve P, then P holds true throughout.
\end{theory}

\begin{theory}{Proof by contradiction}\\
    We can prove that something is true by thinking about its negation. If we assume that
    $\neg P$ is true, and that we somehow prove that this is false, then we can prove that
    $P$ is true. This feels flimsy but it holds true. 

    For any statement $P$, it either has to be true or false. Negating that statement $P$ means
    that $\neq P$ is the opposite of $P$. As such, if $\neq P$ is false, then $P$ is true.
\end{theory}

\begin{example}{Algorithm Reeasoning: Mergesort}\\
    \begin{itemize}
        \item For an algorithm, we need to prove correctness and efficiency
        \item as such, we need to prove that it gives the right answer and jusitfy the
            time complexity
    \end{itemize}

    Lets take mergesort for example:
    We would need to prove that not only mergesort "sorts" correctly, but also it runs in
    $O(nlogn)$ time for a proof to be complete.

    Mergesort takes an array $A[l..r]$, and splits it in half every time. If $l = r$, then
    it simply returns (since there is just one item and theres nothing left to sort).
    \begin{itemize}
        \item Otherwise, you get the midpoint: $(l + r)/2$.
        \item Apply the algorithm recursively to $A[l, m]$ and $A[m+1, r]$
        \item merge the subarrays $A[l..m]$ and $A[m+1..r]$, into a sorted subarray
            $A[l..r]$.
    \end{itemize}

    We need a merge operation to successfully merge the two subarrays $A[l..m]$ and
    $A[m+1..r]$.
    \begin{itemize}
        \item Create a buffer $B$ of size $r - l + 1$. 
        \item fill $B$ from left to right using indices $i, j$ that point to the smallest
            uncopied elements of the two subarrays
        \item copy the smaller of $A[i]$ or $A[j]$ until they go out of bounds
        \item overwrite $A[l..r]$ with $B$ when done.
    \end{itemize}
\end{example}

\begin{example}{Mergesort Proof}\\
    There are two parts we need to prove correctness for: 
    \begin{itemize}
        \item mergesort(A, l, r)
        \item merge(A, l, m, r)
    \end{itemize}

    Lets first prove that merge() successfully creates a sorted array using two sorted
    subarrays:
    \begin{itemize}
        \item Invarient: At any time the items copied into $B$ so far are the smallest elements of
            $A[l..r]$ in sorted order.
        \item Initialisation: This holds since nothing has been copied into $B$
        \item Maintanance: Each iteration copies the smallest item between the two arrays, so the
            invarient holds.
        \item termination: When all elements have been copied, merge terminates.
        \item As such, we can prove that given two sorted subarrays, merge() successfully
            creates a larget sorted array.
    \end{itemize}

    What about the mergesort step?
    Let us use induction.
    \begin{itemize}
        \item Base case: Our array of size 1 is already sorted. 
        \item Inductive assumption: Assume that mergesort correctly sorts all arrays of
            size up to $k$
        \item For an array of size $k + 1$, we can represent it as two split ararys of
            sizes $(k + 1)/2$. 
        \item Running mergesort on the two arrays results in two sorted arrays. 
        \item Running merge on both arrays creates a sorted array of size $k + 1$.
    \end{itemize}

    By induction, the algorithm is correect for all sizes.
\end{example}

\begin{theory}{Greedy Algorithms}\\
    A greedy algorithm is an algorithm that solves a problem by solving it in stages. 

    At each stage, it selects the best possible choice (that it knows of locally). The
    difficult part of greedy problems are actually proving them :c

    The tradeoff for greedy is that although simple to come up and describe, they don't
    consider all (or future) possibilities. So proofs are going to be annoying.
\end{theory}

\begin{aside}{Greedy proofs}\\
    There are two main ways to prove greedy algos:
    \begin{itemize}
        \item Greedy stays ahead
        \item exchange argument
    \end{itemize}

    For these part of the notes, I'll be making use of a combination of 3121 lecture
    slides and \href{https://web.stanford.edu/class/archive/cs/cs161/cs161.1138/handouts/120\%20Guide\%20to\%20Greedy\%20Algorithms.pdf}{stanford's
    guide to greedy algos}
\end{aside}

\begin{aside}{Greedy formats}\\
    Generally, greedy algorithms are used to solve optimisations (minimise/maximise) based
    off a set of constraints.
    \begin{itemize}
        \item Maximise attending events without attending overlapping events
        \item Minimise number of jumps over a pond without falling into the water
        \item Minimise number of edges chosen while maintaining connectivity.
    \end{itemize}

    When we write a greedy proof, we aim for two things:
    \begin{itemize}
        \item Correctness - is it right? does it obey the constraints?
        \item Optimality - is it optimal? Does it maximise/minimise the quantity you want?
    \end{itemize}
\end{aside}

\begin{theory}{Greedy Stays Ahead}\\
    GSA aims to prove that according to some measure, the greedy algorithm is always at
    least as far as the optimal solution during each iteration of the algorithm. We can
    use this established fact to prove that greedy does in fact, stay ahead.

    The typical GSA format is:
    \begin{enumerate}
        \item Solution definition: Define your variables, your algo will produce some $A$
            and gets compared against some $A'$ which is optimal.
        \item Cost function: You want to come up with some cost function that can compare
            both your greedy and optimal solution where you can show the cost of $A$ and
            $A'$ at every step 
        \item Prove GSA: we need to prove that the cost of $A_i$ stays ahead or is not
            lesser than $A'_i$. This is done inductively
        \item Prove optimality: Using the fact that GSA, you can show that greedy is
            optimal. You can usually do this by assuming that greedy isn't optimal, and
            proving that greedy stays ahead to derive a conclusion.
    \end{enumerate}
\end{theory}
\begin{aside}{Challenge of GSA}\\
    \begin{itemize}
        \item cost functions can be difficult to find
        \item often better to prove that if greedy stays ahead according to your cost function, 
            then the algorithm is optimal before showing that greedy actually stays ahead.
        \item another way to think about it is by thinking of greedy as not falling behind
    \end{itemize}
\end{aside}

\begin{example}{GSA: Cell towers}\\
    Let us tackle an example problem: Along the long, straight road from Balladonia (in the West) to Caiguna (in the
    East), houses are scattered quite sparsely, sometimes with long gaps between two consecutive
    houses. Telstra must provide mobile phone service to people who live alongside the road, and
    the range of Telstra’s cell tower is 5km.

    Task: Design an algorithm for placing the smallest number of cell towers alongside the road,
    that is sufficient to cover all houses.

    Let our greedy solution be to place the cell towers 5km to the right of the leftmost
    house.

    Firstly let us claim that for all $k \geq 1$, the $k$th tower in the greedy selection
    is at least as far east as the $k$th tower in the optimal selection. (here we are
    assuming that if greedy stays ahead, then our algorithm is optimal).

    Let us do this inductively. For the first cell tower, (k = 1), greedy places $g_1$ 5km
    to the right of the leftmost house. For the optimal solution, it has to cover at least
    the first house. As such, it can place the cell tower in any interval between $[h_1 -
    5.. h_1 + 5]$, where $h_1$ is the first house. We know that $g_1 = h_1 + 5$. The
    optimal solution cannot place a cell tower further east, since it would not cover $h$.
    As such, $g_1 \geq o_1$.

    Let us assume that this is true for $g_k$ and $o_k$: $$g_k \geq o_k$$

    For tower $k + 1$, we need to prove that $g_{k+1} \geq o_{k+1}$. Given that we know
    the leftmost uncovered house $h'$ exists where $g_k$ and $o_k$ doesn't cover $h'$, we
    know that greedy will place $g_{k+1} -5 = h$. For the sake of contradition, assume
    that $o_{k+1} > g_{k+1}$ while covering h. If $o_{k+1} > g_{k+1}$, then $o_{k+1} > h +
    5$. But that would mean taht $o_{k+1}$ is not covering $h$, so through cotradiction we
    can prove that $g_{k+1} \geq o_{k+1}$
\end{example}

\begin{theory}{Exchange Arguments}\\
    The exchange argument proves for optimality by showing that you can iteratively
    transform any optimal solution into the solution provided by the greedy solution
    without affecting cost and optimality.

    \begin{enumerate}
        \item Define solutions: for a greedy solution $G$ and an optimal solution $O$, so
            define variables explicitly
        \item Compare solns: Show that if $O \neq G$, then they differ in some way (or at
            some point). This could be that things are out of order between $O$ and $G$,
            or that theres a piece in $G$ that is missing in $O$, or something else.
        \item Exchange pieces: Show that you can transform $O$ by changing some piece in
            $O$ for some piece of $G$. this can be order, items, etc, but you need to
            prove that this exchange did not change the optimality of $O$ and yet have a
            different solution.
        \item Iterate: you need to prove/argue that this exchange has lowered the
            diferences between $O$ and $G$ and that you can iteratively do this to $O$ to
            transform it into $G$ without affecting the outcome
    \end{enumerate}
\end{theory}

\begin{aside}{More exchange arguments \& pitfalls}\\
    \begin{itemize}
        \item The greedy solution could be optimal even if $G \neq O$. There can be
            multiple optimal solutions
        \item One thing that helped was thinking of the optimal solution as an "alternate"
            solution that just happened to be optimal.
        \item Your solution may need iterative transformations to turn $O$ into $G$. One
            modification may make them more similar, but not equal. As such, we need to
            end our proofs by showing that we can indeed iterate over $O$ so the
            transformation can happen
        \item Proof must also explain why exchanging a piece $g$ and $o$ can exist in the
            first place. Usually this is trivial but you still need to point out that $O
            \neq G$.
    \end{itemize}
\end{aside}

\begin{example}{Exchange argument: Tape storage}\\
    A list of $n$ files of size $L_i$ have to be stored on a tape. Each file is not
    equally likely to be needed, and one must start at the beginning of the tape and scan
    until they find what they need (linear search).\\

    Task: Order the files on the tape such that the expected retrival time is minimised.

    Claim 1: For a file $f_i$ that has files infront of it, its
    retrieval time would be the sum of all files up to it: $$T_i = (L_1 + ... + L_{i})$$ 

    Claim 2: The expected retrieval time of $F_i$, given that $F_i$ was requested
    would be the product of $p_i$ with $T_i$
    $$E_i = p_i \times T_i = p_i \times (L_1 + ... + L_{i})$$

    Claim 3: For the expected retrieval time of the tape, it would be the sum the expected
    retirieval times of all files: $$\sum_{i=1}^{n} E_i$$

    Aside: Intuitively, we have two variables we need to consider for each file:
    \begin{itemize}
        \item Probability $p_i$ 
        \item Length $L_i$
    \end{itemize}
    An expected outcome of a given set of actions is the average of said actions if it was
    done infinitely. As such, an intuitive way of weighing the cost of a file
    $f_i$ is to consider how many times $L_i$ will be passed over to reach some
    other file compared to how many times $f_i$ is the target file. To justify $f_i$
    occupying a certain space, both metrics matter. The higher $p_i$ is, the closer to the
    front we want it, and the higher $L_i$ is, the further from the front we want it.

    We can represent this pull in opposite directions as the ratio of its probability against its size:
    $$\frac{p_i}{L_i}$$

    Solution: Let us order our files in desending $\frac{p_i}{L_i}$.\\

    Proof: let our greedy solution be $G$, and some optimal solution be $O$. $G_i$ and
    $O_i$ represents the $i$th file in ascending order. Let us assume at some point $G
    \neq O$, and diverges at some $i$ that $G_i \neq O_i$ (since if $G$ = $O$, then greedy
    would be optimal.)

    Let us prove by contradiction. Assume that $O$ is minimised, but at some $O_i$ and
    $O_{i+1}$ that $$\frac{P_i}{L_i} < \frac{P_{i+1}}{L_{i+1}}$$

    Before the swap, the expected retrieval time ($E$) is

    \begin{align*}
        E &= L_1 p_1 + ... \\
          &+ (L_1 + ... + L_i) P_i \\
          &+ (L_1 + ... + L_i + L_{i+1}) P_{i+1}\\
          &+ ... + (L_1 + ... + L_n) + P_n
    \end{align*}

    After swapping $F_i$ and $F_{i+1}$:
    \begin{align*}
        E' &= L_1 p_1 + ... \\
           &+ (L_1 + ... + L_{i+1}) P_{i+1}\\
           &+ (L_1 + ... + L_{i+1} + L_i )P_i \\
           &+ ... + (L_1 + ... + L_n) P_n
    \end{align*}

    As such, 

    $$E - E' = L_i P_{i + 1} - L_{i+1} P_i$$
    Let us rearrange our assumption that 
    $$\frac{P_i}{L_i} < \frac{P_{i+1}}{L_{i+1}} \Rightarrow P_i L_{i+1} < P_{i+1} L_i$$
    As such, 
    $$L_i P_{i + 1} - L_{i+1} P_i > 0$$
    Therefore, $E > E'$. However, our assumption was that 
    $$\frac{P_i}{L_i} < \frac{P_{i+1}}{L_{i+1}}$$, which would imply that $E$ isn't
    optimal. Through proof by contradiction, a larger $\frac{P_i}{L_i}$ should be ordered
    first to minimise expected retrieval time. We can conduct this swap can be done

\begin{theory}{What is dynamic programming}\\
    Dynamic programming is a teechnique where you solve a problem by decomposing it into 
    subproblems that come up multiple
    times. As such, you would solve each subproblem once and store it somewhere so its a
    O(1) access, instead of having to recompute that value.

    The benefits of DP is that you can consider multiple (or all) options at each stage
    without recomputing said stage. More often than not, these subproblems are overlapping
    (occur multiple times).
\end{theory}

\begin{aside}{DP Overview}\\
    Parts of a DP algo:
    \begin{itemize}
        \item A definition of the subproblems
        \item A recurrence relation (how one subproblem leads to smaller
            subproblems)
        \item Base cases, which are the trivial subproblems where recurrences
            aren't needed
    \end{itemize}

    DP execution:
    \begin{itemize}
        \item DP problems are iterative in a bottom up sense. You need to compute smaller
            subproblems (the base cases) before being able to compute larger and larger
            subproblems using the recurrence.
        \item Answers to subproblems will be stored in a lookup table (usually an array
            indexed by the parameters of the subproblem).
    \end{itemize}

    DP format: 
    \begin{itemize}
        \item We need to solve the subproblem in an order which follows the dependencies
            (you need to solve smaller subproblems to solve a subproblem)
        \item The original problem might be a subproblem, or it can be solved by combining
            several subproblems. 
        \item Time complexity is usually given by the number of subproblems multiplied by
            the "average" time taken to solve a subproblem
    \end{itemize}
\end{aside}

\begin{theory}{DP correctness}\\
    To prove that a DP is correct:
    
    \begin{itemize}
        \item The recurrence correectly relates subproblems together
        \item Base cases are correctly solved
        \item Overall answer is correctly solved
    \end{itemize}
\end{theory}
